# V231227、V241216 混合槽位样本、V260523 与 V260629 LK 静态差异

## 结论

官方 OTA 与设备 B 槽实读一致的 V260629 LK 没有改变 XPad2 的 LK A/B 分区
布局。静态证据
显示它保留 fastboot 协议初始化和一组只读/控制命令，同时不再包含标准
`flash:`、`erase:` 命令标记。它更接近受限 fastboot，而不是完全删除 fastboot。

另有两份来自 V241216 当前系统整机 dump 的 LS12 LK，均保留标准 `flash:`、
`erase:`。当前 B 槽已由系统属性和 B 侧签名描述符确认为 V241216 / incremental
`19`；保留旧 A 槽的 incremental `1723478295` 已确认，版本高度疑似 V240813。

版本号已确认的 V260523 LK 同样保留 `flash:`、`erase:`。其官方 OTA payload
补零到 8 MiB 后与设备 A 槽实读镜像完全一致。由此，已知 LS12 LK 移除标准
写入/擦除入口的时间范围可收窄为 V260523 之后、V260629 之前或当版。截至
现有已确认样本，V260523 是最后一个仍保留这两个入口的 LS12 版本。

本报告只基于已授权设备的只读镜像、哈希和静态字符串/结构比较。没有执行
刷写、擦除、切槽或 bootloader patch。

## 样本

| 项目 | V231227 | V260523 | V260629 |
| --- | --- | --- | --- |
| 使用样本 | `lk_a.img` | `lk_a-v260523.img` | `lk_b-v260629.img`（OTA/设备一致） |
| 分区大小 | 8,388,608 bytes | 8,388,608 bytes | 8,388,608 bytes |
| SHA-256 | `a87979a827c005107c68395c88396ce14a418dff0a23f89d473797e1476b3296` | `6ebc4667ef9c0a6a888bda6d020cd744967e966c63b4d0ee6a07e5a21bce3b6a` | `4b5f932dee1d3d6f42a23a4f25c058fae7c7c14488b44d5df0959c6c7252f80e` |
| LK payload size field | 1,044,504 bytes | 1,044,504 bytes | 1,039,712 bytes |
| Build ID | `ls12_mt8797_wifi_64-7003d180-20231215183929-20231227154745` | `ls12_mt8797_wifi_64-dfde152c-20241118095326-20260523165450` | `ls12_mt8797_wifi_64-405e7a01-20260602101307-20260629041106` |

V260629 payload 比旧版小 4,792 bytes。两者都使用 MediaTek 容器头，并带有
`cert1` 签名区域。

V231227 备份中的 `lk_b.img` 哈希为
`2daeb1f36095b44b318410b3f4e8b5d989dcc7bb023d1426c492dab0a3053e74`，
确认是 8 MiB 全零文件，因此不能代表另一份旧 LK。

### V260523 版本与镜像一致性

V260523 官方 A/B OTA metadata 报告 incremental `239`。其中 `lk.img` 为
1,261,568 字节，SHA-256 为
`9e987c2359982f0b2cabbf1e0fb756dd156d3af67f5cb8c423bad3fc9cd2139d`。
将它按 LK 分区格式在末尾补零到 8,388,608 字节，所得 SHA-256 为
`6ebc4667ef9c0a6a888bda6d020cd744967e966c63b4d0ee6a07e5a21bce3b6a`，与
V260629 设备未激活 A 槽实读 `lk_a` 完全相同。内部 Build ID 的最终时间戳也为
2026-05-23，因此这里使用确凿版本名 V260523，而不是观察性推断标签。

### V260629 版本与镜像一致性

LS12 V260629 官方 A/B OTA metadata 报告 incremental `260`，整包 SHA-256 为
`7e91923c7cc28a0174d65022ae4e32e5163f8daf32eebce69bed63da6201c198`。其中
`lk.img` 为 1,257,472 字节，SHA-256 为
`f3286d44b6b2606523244cf066378c5d5f024c233dd82d6b39aae635a2654784`。
补零到 8,388,608 字节后，所得 SHA-256 为
`4b5f932dee1d3d6f42a23a4f25c058fae7c7c14488b44d5df0959c6c7252f80e`，与
两台设备 B 槽实读 `lk_b` 完全相同。内部 Build ID 最终时间戳为 2026-06-29。

该发布样本称为“裁剪版”时，仅指标准 Fastboot `flash:`、`erase:` 注册入口已被
移除；Fastboot 协议初始化、下载、读取和控制命令仍然存在。

### V241216 当前系统与保留旧 A 槽

| 原分区 | SHA-256 | incremental | 内部 Build ID | 版本判断 |
| --- | --- | --- | --- | --- |
| 旧 `lk_a` | `ad8f5ea2b16efd60eb72045b35263b8c290dc5b151d75045e78b2af9a83434bf` | `1723478295` | `ls12_mt8797_wifi_64-45596c65-20240417181119-20240813021105` | 高度疑似 V240813 |
| 当前 `lk_b` | `c87d7cd3903ceccd82a2fb6f4ac127434091ba0e4691d331511e35bb44654419` | `19` | `ls12_mt8797_wifi_64-dfde152c-20241118095326-20241216233205` | V241216 已确认 |

当前系统直接报告 `ro.genie.gota.version=V241216`、
`ro.build.version.incremental=19`，B 侧 boot/vbmeta 签名描述符也均为 `19`。
dump 提供者确认 UI 显示版本为 `V2.4.0`，该人工来源字段与镜像内版本分开记录。
A 侧四份 boot/vbmeta 签名描述符均为 `1723478295`，对应 2024-08-12 23:58:15
CST；这与 `lk_a` 的 2024-08-13 最终构建时间共同支持 V240813，但目前没有
A 侧 `ro.genie.gota.version=V240813` 的直接证据。

## Fastboot 命令面

围绕 fastboot、USB、充电、boot mode、解锁和分区操作筛选出的唯一字符串：

- V231227：510 项
- V260629：506 项
- 两版共有：506 项

V231227 与 V260629 共同保留：

```text
fastboot_init
getvar:
download:
boot
continue
reboot-bootloader
reboot-fastboot
set_active:
max-download-size
partition-size:
partition-type:
is-userspace
```

V231227、V241216 当前 B 槽、保留旧 A 槽和 V260523 均保留、但 V260629 中未观察到：

```text
flash:
erase:
cmd_flash_mmc_sparse_img
emmc_erase
emmc_read
```

Fastboot 命令使用精确 ASCII 命令前缀注册。缺少 `flash:` 和 `erase:` 是该
V260629 LK 未注册标准写入/擦除入口的强证据，但不证明所有底层存储辅助函数
都已移除。它也不证明 USB fastboot 不会枚举；如果设备
屏幕仍显示充电图，主机端 `fastboot devices` 和只读 `getvar` 响应才是协议层
是否存活的判断依据。

## 分区布局

两代观察到的 LK 分区保持一致：

| 分区 | 设备 | 起始扇区 | 扇区数 | 字节 |
| --- | --- | ---: | ---: | ---: |
| `lk_a` | `/dev/block/sdc38` | 1,345,536 | 16,384 | 8,388,608 |
| `lk_b` | `/dev/block/sdc55` | 1,873,920 | 16,384 | 8,388,608 |

因此差异不是 GPT 布局变化，而是 B 槽 bootloader 内容更新。

## 签名与 patch 限制

观察到的 V260629 设备状态为 locked、green Verified Boot，并报告 efuse 已启用。
修改 LK 会改变其签名覆盖内容；即使离线补回命令注册和存储后端，仍需要解决
preloader 对 LK 的认证问题。仓库不提供签名绕过或自动写入代码。
