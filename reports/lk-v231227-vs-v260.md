# LS12 V231227 至 V260629 的五份 LK 静态差异

## 结论

官方 OTA 与设备 B 槽实读一致的 V260629 LK 没有改变 XPad2 的 LK A/B 分区布局。静态证据显示它保留 fastboot 协议初始化和一组只读/控制命令，同时不再注册标准 `flash:`、`erase:`。它更接近受限 fastboot，而不是完全删除 fastboot。

另有两份来自 V241216 当前系统整机 dump 的 LS12 LK，均保留标准 `flash:`、`erase:`。当前 B 槽已由系统属性和 B 侧签名描述符确认为 V241216 / incremental `19`；保留旧 A 槽的 incremental `1723478295` 已确认，版本记为 `V240813（推定）`。

版本号已确认的 V260523 LK 同样保留 `flash:`、`erase:`。其官方 OTA payload 补零到 8 MiB 后与设备 A 槽实读镜像完全一致。由此，已知 LS12 LK 移除标准写入/擦除入口的时间范围可收窄为 V260523 / incremental `239` 之后、V260629 / incremental `260` 之时。截至现有样本，V260523 是最后一份确认仍保留这两个入口的 LS12 版本。

进一步反汇编确认，V260629 删除的是 `flash:`、`erase:` 对外注册调用，而不是全部底层存储后端；`storage_write`、`storage_erase`、`partition_write`、`emmc_write`、`cmd_erase_mmc` 和 UFS/eMMC 写擦路径仍然存在。完整的 Preloader 与 LK 联动时间线见 [启动链演变报告](bootchain-evolution-1703659196-to-260.md)。

本报告只基于已授权设备的只读镜像、哈希和静态字符串/结构比较。没有执行刷写、擦除、切槽或 bootloader patch。

五份分区镜像均为 8,388,608 bytes。V260629 主 payload 比前四份小 4,792 bytes；全部样本都使用 MediaTek 容器头并带有 `cert1` 签名区域。

### 主 payload 变化量

在 payload 大小相同的前四份样本之间，同偏移字节变化为：

| 变化                      | 不同字节 | 占 payload |
| ------------------------- | -------: | ---------: |
| `1703659196 → 1723478295` |       32 |     0.003% |
| `1723478295 → 19`         |       28 |     0.003% |
| `19 → 239`                |      444 |     0.043% |

这些变化主要来自 Build ID、少量常量/指针、调试字符串布局和版本相关认证材料。四份样本的标准 Fastboot 命令字符串及其偏移保持一致。`19` 与 `239` 还使用同一源码修订标识 `dfde152c`，因此可以判断为同一功能基线的不同构建；它们不是逐字节相同的可执行文件。

V231227 备份中的 `lk_b.img` 哈希为 `2daeb1f36095b44b318410b3f4e8b5d989dcc7bb023d1426c492dab0a3053e74`，确认是 8 MiB 全零文件，因此不能代表另一份旧 LK。

### V260523 版本与镜像一致性

V260523 官方 A/B OTA metadata 报告 incremental `239`。其中 `lk.img` 为 1,261,568 字节，SHA-256 为 `9e987c2359982f0b2cabbf1e0fb756dd156d3af67f5cb8c423bad3fc9cd2139d`。将它按 LK 分区格式在末尾补零到 8,388,608 字节，所得 SHA-256 为 `6ebc4667ef9c0a6a888bda6d020cd744967e966c63b4d0ee6a07e5a21bce3b6a`，与 V260629 设备未激活 A 槽实读 `lk_a` 完全相同。内部 Build ID 的最终时间戳也为 2026-05-23，因此这里使用确凿版本名 V260523，而不是观察性推断标签。

### V260629 版本与镜像一致性

LS12 V260629 官方 A/B OTA metadata 报告 incremental `260`，整包 SHA-256 为 `7e91923c7cc28a0174d65022ae4e32e5163f8daf32eebce69bed63da6201c198`。其中 `lk.img` 为 1,257,472 字节，SHA-256 为 `f3286d44b6b2606523244cf066378c5d5f024c233dd82d6b39aae635a2654784`。补零到 8,388,608 字节后，所得 SHA-256 为 `4b5f932dee1d3d6f42a23a4f25c058fae7c7c14488b44d5df0959c6c7252f80e`，与两台设备 B 槽实读 `lk_b` 完全相同。内部 Build ID 最终时间戳为 2026-06-29。

该发布样本称为“裁剪版”时，指标准 Fastboot `flash:`、`erase:` 注册入口已被移除；Fastboot 协议初始化、下载、读取和控制命令仍然存在，底层存储写擦代码也没有被整体删除。

### V241216 当前系统与保留旧 A 槽

| 原分区 | SHA-256 | incremental | 内部 Build ID | 版本判断 |
| --- | --- | --- | --- | --- |
| 旧 `lk_a` | `ad8f5ea2b16efd60eb72045b35263b8c290dc5b151d75045e78b2af9a83434bf` | `1723478295` | `ls12_mt8797_wifi_64-45596c65-20240417181119-20240813021105` | V240813（推定） |
| 当前 `lk_b` | `c87d7cd3903ceccd82a2fb6f4ac127434091ba0e4691d331511e35bb44654419` | `19` | `ls12_mt8797_wifi_64-dfde152c-20241118095326-20241216233205` | V241216 已确认 |

当前系统直接报告 `ro.genie.gota.version=V241216`、`ro.build.version.incremental=19`，B 侧 boot/vbmeta 签名描述符也均为 `19`。dump 提供者确认 UI 显示版本为 `V2.4.0`，该人工来源字段与镜像内版本分开记录。A 侧四份 boot/vbmeta 签名描述符均为 `1723478295`，对应 2024-08-12 23:58:15 CST；这与 `lk_a` 的 2024-08-13 最终构建时间共同支持 `V240813（推定）`，但目前没有 A 侧 `ro.genie.gota.version=V240813` 的直接证据。

## Fastboot 命令面

| 命令或能力 | V231227 | 2024-08-13 | V241216 | V260523 | V260629 |
| --- | :-: | :-: | :-: | :-: | :-: |
| `fastboot_init` | 是 | 是 | 是 | 是 | 是 |
| `getvar:` | 是 | 是 | 是 | 是 | 是 |
| `download:` | 是 | 是 | 是 | 是 | 是 |
| `boot` / `continue` | 是 | 是 | 是 | 是 | 是 |
| `reboot-bootloader` / `reboot-fastboot` | 是 | 是 | 是 | 是 | 是 |
| `set_active:` | 是 | 是 | 是 | 是 | 是 |
| `flash:` | 是 | 是 | 是 | 是 | **否** |
| `erase:` | 是 | 是 | 是 | 是 | **否** |

Fastboot 使用精确 ASCII 命令前缀注册。旧版 Thumb 反汇编可定位到 `flash:`、`erase:` 被传给命令注册函数的两次调用；V260629 中相应字符串和注册调用均不再存在。这比单纯比较字符串更直接地证明标准写入/擦除入口被移除。

V260629 同时缺少 `cmd_flash_mmc_sparse_img`、`emmc_erase`、`emmc_read` 等部分直接包装符号，但仍保留：

```text
storage_write
storage_erase
partition_write
emmc_write
cmd_erase_mmc
稀疏镜像解析和写入错误路径
UFS / eMMC 底层写入与擦除路径
```

V260629 的 Fastboot 帮助文本甚至仍列出 `flash xxpartid xxx.img` 与 `erase xxpartid`。这些残留文本不能代替真实命令注册，也说明本次变化更接近接口层策略裁剪，不是完整移除存储子系统。`download:` 仍在只表示可以向 RAM 接收数据，不代表存在把数据持久化到分区的标准 Fastboot 命令。

V260523 与 V260629 围绕充电、显示、USB 和 boot mode 的静态字符串面没有观察到对应功能项删除。标准写入命令缺失本身也不证明 USB fastboot 不会枚举；如果屏幕仍显示充电图，应使用主机端 `fastboot devices` 和只读 `getvar` 响应判断协议层是否存活。

## 签名与 DTB 连续性

五份 LK 主镜像的 `cert1` DER 完全相同，其 SHA-256 均为：

```text
91e42388da7168ab01e695d2e696690ef2757e4edf26ea7216b584695ae021c5
```

五份 `cert2` 内容会随镜像变化，但 RSA 公钥模数相同，且仍由 `cert1` 中的第二把公钥授权。独立封装的 `lk_main_dtb` 也在五份样本中逐字节相同：

```text
SHA-256 = af3efe2d5bf158b83bbe31cb73f3123848d4784a1dc5df7f7e87eadaf4a80a91
```

因此 V260629 的功能收紧不是由 LK 信任根轮换、A/B 分区变化或主 DTB 更换造成。

## 裁剪窗口与 2025 样本缺口

V260523 / incremental `239` 构建于 2026-05-23，仍注册 `flash:`、`erase:`；V260629 / incremental `260` 构建于 2026-06-29，已移除两者。当前已确认窗口为 37 天。

缺少 2025 年若干 dump 不影响这个窗口，因为 `239` 本身已经晚于 2025 年并仍保留完整命令入口。但缺失样本意味着不能证明 2025 每份内部或公开版本都相同，也不能排除中间版本曾改变后又恢复。因此“V260523 是最后一版”应限定为“当前最后一份已确认样本”。

## 分区布局

两代观察到的 LK 分区保持一致：

| 分区   | 设备               |  起始扇区 | 扇区数 |      字节 |
| ------ | ------------------ | --------: | -----: | --------: |
| `lk_a` | `/dev/block/sdc38` | 1,345,536 | 16,384 | 8,388,608 |
| `lk_b` | `/dev/block/sdc55` | 1,873,920 | 16,384 | 8,388,608 |

因此差异不是 GPT 布局变化，而是 B 槽 bootloader 内容更新。

## 签名与 patch 限制

观察到的 V260629 设备状态为 locked、green Verified Boot，并报告 efuse 已启用。修改 LK 会改变其签名覆盖内容；即使离线补回命令注册和存储后端，仍需要解决 preloader 对 LK 的认证问题。仓库不提供签名绕过或自动写入代码。
